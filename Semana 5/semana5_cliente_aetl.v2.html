📚 Fundamentos Teóricos AETL — Semana 5

Los 5 constructos del modelo, aplicados al contexto de evaluación y consolidación

🧩 1. Constructivismo — La Evaluación como Construcción de Conocimiento
Piaget (1978); Vygotsky (1978); Bognár (2024)

El examen práctico no es un "test de memoria": es una situación donde el estudiante construye activamente un cliente en un contexto nuevo, usando los esquemas mentales que desarrolló en semanas anteriores. Vygotsky subraya que la demostración de competencia en la Zona de Desarrollo Próximo — sin el andamiaje de ejemplos vistos — es evidencia real de aprendizaje.

Constructo operacional esta semana: Usa la IA hoy para consolidar tu comprensión de los patrones — no para que los implemente por ti. El examen evalúa tu capacidad de reconstruir el patrón en un contexto nuevo, lo que requiere comprensión profunda, no memorización.
📋 2. Aprendizaje Autorregulado (SRL) — Diagnóstico Activo Previo al Examen
Zimmerman (2000); Chiu (2024); Sari et al. (2025)

El SRL establece que el aprendizaje efectivo incluye la fase de autoevaluación: identificar qué sabes bien y qué tienes débil antes del examen, no después. El ciclo completo es: planificación (¿qué necesito repasar?) → ejecución (simular el examen) → reflexión (¿qué gaps encontré?).

Constructo operacional esta semana: El Reto 1 en COMPRENDE es un diagnóstico activo: formúlale a la IA preguntas específicas sobre los conceptos donde tienes más duda y mide con honestidad cuáles puedes explicar con tus propias palabras.
⚖️ 3. Teoría de la Carga Cognitiva — Preparar sin Saturar
Sweller (1988/2020); Mahdi et al. (2025)

En una semana de evaluación, el riesgo es la carga extrínseca por pánico: querer repasar todo al mismo tiempo, lo que satura la memoria de trabajo sin consolidar nada. La estrategia óptima es reducir la carga extrínseca (no revisar 4 semanas en paralelo) y concentrar la carga germana en los patrones que el examen evalúa: construcción de petición → manejo de respuesta → manejo de error.

Constructo operacional esta semana: No repases sintaxis — repasa decisiones. ¿Por qué timeout=10 y no timeout=60? ¿Por qué no reintentar un 4xx? ¿Por qué usar Observer en lugar de llamar directamente a la UI? Esas decisiones son las que se evalúan.
🤝 4. Agencia Compartida — Tú eres el Arquitecto, la IA es el Revisor
Bandura (2001); Brod et al. (2023)

Esta semana, la agencia compartida toma una forma particular: la IA actúa como evaluador crítico de tus respuestas, no como proveedor de respuestas. Cuando hagas el simulacro del examen (Reto 2), tú propones el código y la IA señala lo que falta o está mal — exactamente como un revisor técnico senior en la industria.

Constructo operacional esta semana: En el simulacro, escribe primero, pide retroalimentación después. Si le pides a la IA el código completo antes de intentarlo, estás renunciando a tu rol de Arquitecto del Cliente — y el examen real no tendrá IA que lo haga por ti.
🌐 5. Taxonomía de Bloom — Evaluar = Crear bajo Restricciones
Anderson et al. (2001)

El Examen Práctico 1 opera en los niveles más altos de Bloom aplicados al código cliente: Aplicar (implementar el patrón en un contexto nuevo), Analizar (identificar qué falta en el manejo de errores), y Evaluar (justificar las decisiones de diseño). No es un examen de nivel "Recordar" — no hay código para memorizar.

Constructo operacional esta semana: La rúbrica del examen tiene criterios explícitos de Aplicar y Evaluar. Antes del examen, asegúrate de poder explicar oralmente cada decisión de diseño de tu cliente: ¿por qué este timeout? ¿por qué esta estructura de errores? ¿qué pasaría si eliminas el Observer?

Pregunta de diagnóstico: Si llamas a una función async sin await, ¿qué obtienes? Si la respuesta es "un objeto coroutine que nunca se ejecuta" — dominas el concepto. Si la respuesta es "no sé" — repasa el event loop antes del examen.

📡 Bloque 3: Polling Adaptativo — Detección de Cambios Semana 4

Lo que debes poder hacer de memoria: Implementar un ciclo de polling con intervalo adaptativo, usar ETag/If-None-Match para detectar cambios sin reprocesar datos idénticos, y detener limpiamente el ciclo cuando el usuario lo solicita.

Pseudocódigo del ciclo de polling: constantes INTERVALO_BASE de 5 segundos e INTERVALO_MAX de 60 segundos. Ciclo while ejecutando: hace petición GET con If-None-Match. Si respuesta 200 notifica observadores y resetea intervalo. Si 304 incrementa intervalo (backoff). Si 503 incrementa intervalo. Si 4xx registra error sin reintentar. Si excepción de red registra warning y continúa. Al final espera asyncio.sleep con el intervalo calculado. ┌──────────────────────────────────────────────────────────────┐ │ PSEUDOCÓDIGO DE REFERENCIA — POLLING ADAPTATIVO │ ├──────────────────────────────────────────────────────────────┤ │ │ │ INTERVALO_BASE = 5s // Espera mínima entre consultas │ │ INTERVALO_MAX = 60s // Espera máxima (evita consultas │ │ // innecesarias cuando no hay cambios │ │ ejecutando = True // Bandera de control del ciclo │ │ ultimo_etag = None // Último ETag conocido del servidor │ │ │ │ async función ciclo_polling(): │ │ intervalo = INTERVALO_BASE │ │ mientras ejecutando: │ │ intentar: │ │ headers = {} │ │ si ultimo_etag: │ │ headers["If-None-Match"] = ultimo_etag // ← clave │ │ resp = await GET /inventario headers=headers │ │ │ │ si resp.status == 304: // No cambió │ │ intervalo = min(intervalo * 1.5, INTERVALO_MAX) │ │ // → esperar más tiempo si no hay novedades │ │ si resp.status == 200: // Cambió │ │ ultimo_etag = resp.headers["ETag"] │ │ datos = resp.json() │ │ notificar_observadores(datos) // patrón Observer │ │ intervalo = INTERVALO_BASE // resetear espera │ │ si resp.status >= 500: // Error servidor │ │ intervalo = min(intervalo * 2, INTERVALO_MAX) │ │ // → backoff más agresivo ante errores │ │ capturar Timeout: │ │ registro.warning("Timeout en polling") │ │ intervalo = min(intervalo * 2, INTERVALO_MAX) │ │ await sleep(intervalo) // esperar SIN bloquear │ │ │ │ función detener(): │ │ ejecutando = False // el ciclo termina en la próxima │ │ // iteración — "cierre suave" │ └──────────────────────────────────────────────────────────────┘
Pregunta de diagnóstico: ¿Por qué el intervalo se resetea a INTERVALO_BASE cuando llega un 200 con cambios? Porque si hay cambios frecuentes, queremos consultar más seguido — el backoff existe para reducir consultas innecesarias, no para castigar la actividad real del servidor.

🔒 Invariantes que no deben romperse — Semana 5 (repaso)

  • El cliente nunca debe crashear silenciosamente ante un error HTTP 5xx — siempre registra, siempre retorna un estado conocido.
  • Ninguna petición HTTP se envía sin un timeout definido — una petición sin timeout puede congelar el cliente indefinidamente.
  • Un error 4xx nunca se reintenta automáticamente — el cliente está enviando algo incorrecto; reintentarlo no lo arregla.
  • El Observable nunca debe hacer referencia directa a una implementación específica de observador — solo llama al método actualizar() de la interfaz.
  • El ciclo de polling siempre debe tener una bandera de control (ejecutando) que permita detenerlo limpiamente sin cancelación forzada.

⚠️ Errores Comunes en el Simulacro (y en el Examen)

Error 1: Manejar todos los errores igual

Hacer un solo except Exception que imprime "Error" para cualquier falla — sin diferenciar TimeoutError, ConnectionError, y errores HTTP. En el examen, esto se evalúa: cada tipo de error necesita una respuesta diferente en el cliente.

Error 2: Olvidar el 304 — comparar siempre contra el último estado

Detectar cambios comparando el body completo (if datos != ultimo_datos) en lugar de usar ETag. Funciona, pero descarta la optimización central de la Semana 4: el servidor ya sabe si cambió — pregúntale con If-None-Match en lugar de procesar el body tú mismo.

Error 3: Polling sin backoff — intervalo fijo aunque no haya cambios

Usar await sleep(INTERVALO_BASE) siempre, sin aumentar el intervalo cuando hay respuestas 304 consecutivas. El backoff adaptativo reduce las conexiones abiertas en el cliente y el tráfico de red cuando el inventario está estático.

Error 4: Observer que sabe demasiado — acoplamiento implícito

Hacer que _notificar() pase diferentes tipos de datos a diferentes observadores según su tipo (if isinstance(obs, ObservadorUI): notificar_con_formato()). El Observable debe tratar a todos los observadores de forma idéntica — la interfaz actualizar(datos) es el único contrato.

🔒 Invariantes del Simulacro — Verifica en tu implementación

  • _consultar_pedidos() nunca lanza una excepción no capturada — el ciclo de polling no debe detenerse por un fallo puntual de red.
  • Un 503 del servidor incrementa el intervalo (backoff); un 400 del servidor registra un error pero NO reintenta — algo en la petición está mal.
  • iniciar() es async y usa await asyncio.sleep() (no time.sleep() bloqueante) para no congelar el event loop.
  • detener() solo cambia una bandera — no cancela la corrutina de forma forzada ni genera CancelledError en el ciclo.

🚫 Condiciones de Fallo Crítico — Zero Tolerance

En sistemas distribuidos del lado del cliente, ciertos fallos son independientemente fatales: no importa qué tan bien estructurado esté el resto del código. Si se produce cualquiera de las siguientes condiciones, la calificación máxima para el criterio afectado queda limitada a Básico (50%), sin excepción:

🔴 Crash silencioso que detiene el event loop
El cliente recibe un error de red (Timeout, ConnectionError) y lanza una excepción no capturada que termina la corrutina principal. El ciclo de polling muere sin registrar el error ni notificar a los observadores. Aplica al criterio: Manejo de errores (25%) + Programación asíncrona (20%)
🔴 Excepción no capturada en actualizar() de un observador detiene la notificación
Si el Observable no protege el ciclo for obs in self._observadores con un try/except, un observador con bug silencia a todos los observadores siguientes sin que el sistema lo reporte. Aplica al criterio: Patrón Observer (15%)
🔴 time.sleep() bloqueante en el ciclo de polling
Bloquea el event loop completo durante el intervalo de espera: ninguna otra corrutina puede ejecutarse, peticiones concurrentes quedan congeladas, y el cliente no puede responder a detener(). Aplica al criterio: Programación asíncrona (20%)

Nota: un cierre controlado del ciclo (ej: ejecutando = False + salir del while limpiamente) es correcto y no activa estas condiciones. La distinción es fallo silencioso vs cierre controlado.

Hacia la Semana 6: Arquitectura en Capas

🔮 ¿Qué viene la próxima semana?

La Semana 6 introduce la arquitectura en capas del cliente: MVC, Repository Pattern, separación de responsabilidades. Si tu cliente actual mezcla la lógica HTTP con la lógica de negocio (ej: el archivo que hace peticiones también procesa los datos y los muestra), la Semana 6 será más difícil.

¿Por qué el Observer de Semana 4 es prerequisito de la Semana 6? Porque el Repository Pattern de la Semana 6 es el Observer aplicado a la capa de datos: el repositorio es el Observable; la vista y la lógica de negocio son los observadores. Si no entiendes el desacoplamiento del Observer, el Repository se sentirá como burocracia sin propósito.

Preparación recomendada: Revisa tu código del Hito 1 y pregúntate: ¿Dónde mezclo responsabilidades? ¿Hay una función que hace HTTP, procesa datos Y los muestra? Esas son las candidatas a separar en la Semana 6.

Idempotencia desde el cliente: Si tu cliente hace POST /pedido y no recibe respuesta (timeout), ¿fue creado o no? La idempotencia del cliente implica diseñar la petición de forma que puedas reintentarla sin duplicar el pedido — enviando un ID único generado por el cliente (idempotency key).

📖 Glosario — Semana 5

Examen Práctico
Evaluación en vivo donde el estudiante implementa un cliente distribuido ante un escenario nuevo, demostrando transferibilidad de los patrones aprendidos.
Rúbrica (Rubric)
Herramienta de evaluación con criterios y niveles de desempeño explícitos. Permite al estudiante saber exactamente qué se evalúa y a qué nivel debe llegar.
Transferibilidad
Capacidad de aplicar un patrón o concepto aprendido en un contexto a uno diferente. Ej: aplicar el ciclo polling+Observer a un endpoint de pedidos aunque lo aprendiste con inventario.
Backoff Exponencial
Estrategia donde el intervalo de espera entre reintentos se multiplica en cada intento (ej: 1s, 2s, 4s, 8s). Reduce la carga acumulada de conexiones en el cliente y el tráfico de red.
Jitter (Variación Aleatoria)
Aleatoriedad añadida al backoff para evitar que múltiples instancias del cliente se sincronicen y reconecten simultáneamente. Variante más efectiva: jitter decorrelacionado.
Thundering Herd
Problema donde muchas instancias del cliente reintentan simultáneamente después de un fallo compartido. Cada cliente experimenta más rechazos y timeouts porque todos compiten por los mismos recursos del servidor al mismo tiempo.
Idempotency Key
Identificador único generado por el cliente e incluido en una petición POST, que permite al servidor ignorar duplicados si el cliente reintenta la petición ante un timeout. El cliente puede reintentar de forma segura.
Circuit Breaker (Interruptor del Circuito)
Patrón del lado cliente donde, después de N fallos consecutivos, el cliente deja de intentar contactar al servidor por un período, evitando desperdiciar recursos en conexiones que saben que fallarán. Se implementará en Semana 9.
Repository Pattern
Patrón arquitectónico que abstrae el origen de datos (API, caché local, base de datos) detrás de una interfaz uniforme. El cliente solicita datos al repositorio sin saber si vienen de la red o del caché. Se verá en Semana 6.
Architecture Decision Record (ADR)
Documento breve que registra una decisión arquitectónica significativa, su contexto, las alternativas consideradas y las consecuencias de la elección. El docstring de decisiones de diseño de este curso es un ADR informal.
Agencia Compartida (AETL)
Balance entre autonomía del estudiante y asistencia de la IA. Esta semana toma forma de Evaluador: el estudiante propone soluciones, la IA señala errores sin resolverlos, preservando el aprendizaje activo.
Caso Frontera (Edge Case)
Escenario límite o inusual que puede revelar fallos en el cliente. En el examen: null en el body, Content-Type inesperado, 304 sin ETag previo, timeout justo en el límite configurado.
