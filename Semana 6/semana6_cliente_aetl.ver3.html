🎬 Escenario: TechNova — Panel de Alertas de EcoMarket en Tiempo Real

EcoMarket acaba de lanzar un panel de operaciones que muestra en vivo: cambios de precio de productos, alertas de stock crítico y notificaciones de nuevos pedidos. El equipo de TechNova implementó esta funcionalidad con polling cada 3 segundos.

🔗 Conexión con semanas anteriores

En la Semana 4 construiste un cliente que pregunta al servidor cada N segundos (polling). Esta semana el flujo se invierte: el servidor empuja datos al cliente a través de una conexión HTTP que permanece abierta. El cliente asíncrono que ya tienes (con timeouts y manejo de errores) sigue siendo necesario — ahora ese "ciclo de espera" sucede dentro de la conexión SSE, no entre peticiones repetidas.

¿Qué es Server-Sent Events?

SSE es un estándar HTML5 que permite al servidor enviar datos al cliente a través de una conexión HTTP unidireccional persistente. El cliente abre una sola conexión y el servidor la mantiene viva, enviando mensajes cuando los datos cambian. No hay peticiones repetidas — solo una conexión que fluye.

╔══════════════════════════════════════════════════════════════════════════╗ ║ POLLING (Semana 4) vs SERVER-SENT EVENTS (Semana 6) ║ ╠══════════════════════════════════════════════════════════════════════════╣ ║ ║ ║ POLLING — el cliente pregunta ║ ║ ───────────────────────────── ║ ║ Cliente Servidor ║ ║ │──GET /api/productos──▶│ t=0s: petición #1 (200 OK, datos) ║ ║ │──GET /api/productos──▶│ t=3s: petición #2 (304 sin cambios) ║ ║ │──GET /api/productos──▶│ t=6s: petición #3 (304 sin cambios) ║ ║ │──GET /api/productos──▶│ t=9s: petición #4 (200 OK, cambio!) ║ ║ │ │ 4 conexiones TCP, 75% de respuestas vacías ║ ║ ║ ║ SERVER-SENT EVENTS — el servidor empuja ║ ║ ─────────────────────────────────────── ║ ║ Cliente Servidor ║ ║ │──GET /eventos──────▶ │ t=0s: abre conexión (200 + stream) ║ ║ │◀── data: precio=45.00 │ t=2s: server envía evento (hay cambio) ║ ║ │◀── data: stock=0 │ t=11s: server envía evento (agotado) ║ ║ │◀── data: precio=43.00 │ t=14s: server envía evento (descuento) ║ ║ │ │ 1 conexión TCP, 0 peticiones vacías ║ ╚══════════════════════════════════════════════════════════════════════════╝
El diagrama muestra la diferencia estructural entre polling y SSE. En polling, cada ciclo genera una conexión TCP nueva — la mayoría sin datos útiles. En SSE, una sola conexión permanece abierta y el servidor envía mensajes solo cuando hay cambios. El ahorro de red y CPU del servidor es proporcional al número de clientes concurrentes.

El protocolo SSE: formato de mensajes

Los mensajes SSE son texto plano con un formato muy simple. Cada mensaje tiene campos opcionales y termina con una línea en blanco adicional (\n\n):

╔══════════════════════════════════════════════════════════════════╗ ║ FORMATO DE UN MENSAJE SSE ║ ╠══════════════════════════════════════════════════════════════════╣ ║ ║ ║ id: 42 ← identificador del evento ║ ║ event: precio-actualizado ← tipo de evento (opcional) ║ ║ data: {"producto":"A01","precio":45.00,"moneda":"MXN"} ║ ║ retry: 3000 ← ms antes de reconectar (opcional) ║ ║ ← línea en blanco → fin del mensaje ║ ║ ║ ║ id: 43 ← siguiente mensaje ║ ║ event: stock-critico ║ ║ data: {"producto":"B07","stock":2,"umbral":5} ║ ║ ║ ║ REGLAS: ║ ║ • Cada campo: "nombre: valor\n" ║ ║ • data multilínea: múltiples líneas data: ║ ║ • Mensaje vacío (solo \n\n): comentario de keep-alive ║ ║ • Content-Type del servidor: text/event-stream ║ ╚══════════════════════════════════════════════════════════════════╝
El formato SSE es deliberadamente simple: texto plano, sin XML ni JSON a nivel de protocolo (el JSON va dentro del campo data). Esta simplicidad facilita el diagnóstico desde consola y la lectura manual de logs. El campo id es clave para la reconexión: el cliente lo almacena y lo envía como Last-Event-ID al reconectarse, permitiendo al servidor reenviar eventos perdidos.

La API EventSource (navegador) y el cliente manual (Python/Node)

En navegadores, la API EventSource gestiona automáticamente la reconexión y los headers. En Python o Node.js sin framework, el cliente debe leer el stream línea a línea y construir los mensajes manualmente:

Aspecto EventSource (navegador) Cliente manual (Python/Node)
Conexión Automática al instanciar El desarrollador abre la petición HTTP
Reconexión Automática con Last-Event-ID El desarrollador debe implementarla
Parseo del stream Automático Leer línea a línea, construir mensajes
Eventos personalizados addEventListener('tipo', cb) El desarrollador detecta el campo event:
Headers de autenticación No soportados directamente Flexibles — incluir en la petición
CORS Sujeto a política del navegador No aplica (servidor a servidor)
╔══════════════════════════════════════════════════════════════════════╗ ║ FLUJO DE RECONEXIÓN — Cómo el cliente no pierde eventos ║ ╠══════════════════════════════════════════════════════════════════════╣ ║ ║ ║ Cliente Servidor ║ ║ │ │ ║ ║ │──GET /eventos─────────────────▶ │ conexión inicial ║ ║ │◀── id:10 data:precio=45 │ evento 10 ║ ║ │◀── id:11 data:stock=3 │ evento 11 ← almacenado ║ ║ │ │ ║ ║ │ ✗ CORTE DE RED │ ║ ║ │ ⏳ espera retry (3s por defecto) │ ║ ║ │ │ ║ ║ │──GET /eventos─────────────────▶ │ ║ ║ │ Last-Event-ID: 11 │ ← el servidor sabe desde ║ ║ │ │ dónde reanudar ║ ║ │◀── id:12 data:stock=0 │ evento 12 (no se perdió 12) ║ ║ │◀── id:13 data:pedido=nuevo │ evento 13 ║ ║ ║ ║ SIN Last-Event-ID → el servidor no sabe qué perdiste ║ ║ CON Last-Event-ID → el servidor puede reenviar desde id+1 ║ ╚══════════════════════════════════════════════════════════════════════╝
El diagrama ilustra el mecanismo de recuperación ante fallos de red. El campo id en cada mensaje permite al cliente llevar la cuenta del último evento recibido. Al reconectarse, el header Last-Event-ID comunica este valor al servidor. El servidor decide cuánto tiempo conserva el historial — si no lo conserva, el cliente solo recibirá eventos futuros.

🔒 Invariantes que no deben romperse — Fase COMPRENDE

  • Un mensaje SSE termina siempre con una línea en blanco (\n\n). Sin ella, el cliente no sabe que el mensaje está completo.
  • El Content-Type del servidor debe ser text/event-stream. Si el servidor responde con application/json, no es SSE.
  • El campo id es opcional — pero sin él, el cliente no puede pedir al servidor reenviar eventos perdidos.
  • SSE es unidireccional: servidor → cliente. Si necesitas enviar datos del cliente al servidor mientras la conexión SSE está abierta, necesitas una petición HTTP separada.
  • Un comentario SSE (: texto) no es un evento — es un keep-alive. El cliente lo ignora, pero mantiene la conexión viva.

🔒 Invariantes que no deben romperse — Fase APLICA

  • Nunca procesar un mensaje SSE incompleto — esperar siempre la línea en blanco que lo termina.
  • El buffer de líneas debe resetearse completamente después de cada mensaje procesado — acumular líneas de mensajes anteriores causa parseo incorrecto.
  • El campo retry: actualiza retry_ms — si el servidor lo envía, el cliente debe respetarlo, no usar el default inamovible.
  • Una excepción al procesar un evento no debe cerrar la conexión — solo loguear el error y continuar leyendo el stream.
  • Después de detener(), no debe quedar ninguna tarea o corutina activa.
  • Si el servidor responde 204 No Content, el cliente no reconecta — es la señal de que el stream terminó intencionalmente.

⚠️ El problema silencioso: ¿qué pasa si el historial del servidor se purga?

El mecanismo Last-Event-ID asume que el servidor conserva un buffer de eventos recientes. En sistemas distribuidos reales, ese buffer tiene un límite — si el cliente estuvo desconectado demasiado tiempo, el servidor ya no tiene los eventos intermedios.

Cuando el cliente reconecta con un Last-Event-ID antiguo, el servidor puede responder de dos formas: (a) enviar eventos desde el punto solicitado si aún los tiene, o (b) enviar un evento especial event: reset indicando que el historial se perdió. En ese caso, el cliente distribuido robusto debe saber que su estado local es inválido y ejecutar un Full Fetch — una petición GET tradicional completa para resincronizar el estado — antes de volver a escuchar el stream.

No necesitas implementar esto esta semana — pero debes saber que existe. La Semana 7 abordará estrategias de recuperación de estado bajo SSE con historial limitado.

⚠️ Errores comunes esta semana

Confundir SSE con WebSocket: SSE es unidireccional sobre HTTP estándar — no requiere upgrade de protocolo ni handshake especial. Si la IA te genera código con ws:// o websocket, está resolviendo un problema diferente.
Procesar líneas en lugar de mensajes: Un mensaje SSE puede tener múltiples líneas data: (datos multilínea). Si procesas cada línea data: como un evento independiente, estás partiendo mensajes a la mitad.
Reconexión sin límite de intentos: La IA tiende a generar bucles while True: reconectar() sin backoff ni límite. Ante una caída del servidor, esto genera una tormenta de reconexiones que agrava el problema.
Ignorar el campo retry: del servidor: El servidor puede enviar retry: 5000 para indicar que el cliente espere 5 segundos antes de reconectar. Si el cliente usa un valor hardcodeado, ignora esta señal de control del servidor.
🚨 Trampa de HTTP/1.1 — el límite de 6 conexiones por dominio (solo Ruta B / navegador): Los navegadores bajo HTTP/1.1 permiten como máximo 6 conexiones TCP simultáneas al mismo dominio. Si abres 6 pestañas del panel EcoMarket con EventSource, la séptima (o cualquier petición REST al mismo servidor) quedará en estado pending indefinidamente — bloqueada sin error visible. Síntoma: las peticiones normales de la app se congelan sin razón aparente. Causa raíz: las 6 conexiones SSE ocupan todos los slots disponibles. Solución: HTTP/2 (que multiplexa conexiones) o agrupar múltiples tipos de eventos en un solo stream SSE. Esto es exactamente lo que abordará la Semana 7.

🔒 Invariantes que no deben romperse — Fase REFLEXIONA

  • SSE no reemplaza polling en todos los casos — solo cuando el servidor puede generar el stream y el flujo es unidireccional.
  • Una conexión SSE persistente bajo HTTP/1.1 ocupa un slot de conexión del navegador — con HTTP/2, este costo desaparece gracias a la multiplexación.
  • La decisión SSE vs polling debe justificarse con datos del cliente (número de conexiones, peticiones vacías, latencia) — no solo con "SSE es más moderno".

📖 Glosario — Semana 6

Server-Sent Events (SSE)
Estándar HTML5 para comunicación unidireccional servidor → cliente sobre una conexión HTTP persistente. El servidor envía mensajes solo cuando hay datos nuevos.
EventSource
API del navegador que implementa SSE automáticamente: maneja la conexión, el parseo del stream y la reconexión. En Python/Node.js, el desarrollador la implementa manualmente.
text/event-stream
MIME type que identifica un stream SSE. El servidor debe devolver este Content-Type para que el cliente trate la respuesta como un stream de eventos.
Last-Event-ID
Header HTTP que el cliente envía al reconectarse, indicando el ID del último evento recibido. Permite al servidor reanudar el stream sin reenviar eventos ya procesados.
retry (campo SSE)
Campo opcional en un mensaje SSE que indica cuántos milisegundos debe esperar el cliente antes de reconectar si la conexión se pierde. Sobrescribe el valor por defecto del cliente.
Keep-alive (SSE)
Comentario SSE (: texto) enviado por el servidor para mantener la conexión viva cuando no hay eventos. El cliente lo ignora como dato pero el TCP lo procesa para evitar timeouts de red.
Push vs Pull
Pull: el cliente solicita datos periódicamente (polling). Push: el servidor envía datos cuando los genera (SSE, WebSocket). La elección depende de la frecuencia de cambio y la arquitectura del servidor.
Stream chunked
Mecanismo HTTP donde la respuesta se envía en partes (chunks) sin un Content-Length definido. SSE usa Transfer-Encoding: chunked para mantener la conexión abierta indefinidamente.
Andamiaje cognitivo (AETL)
Soporte temporal que la IA proporciona para que el estudiante llegue a comprensiones que aún no alcanza de forma independiente — retirado gradualmente a medida que el estudiante domina el tema.
Auditoría de IA (AETL)
Proceso de revisión crítica del código generado por IA para detectar errores que superan las pruebas superficiales pero fallan en producción. Metacognición aplicada al uso de herramientas IA.
