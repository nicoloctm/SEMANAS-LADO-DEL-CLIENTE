¿Qué es un JWT y qué puede ver el cliente?

Un JSON Web Token (JWT) es un objeto de seguridad codificado en Base64URL, dividido en tres partes separadas por puntos (.). Es emitido por el servidor después de autenticar al usuario y entregado al cliente. Desde ese momento, el cliente es su custodio.

🚨 Concepto crítico: el cliente NO verifica la firma

El payload del JWT está en Base64URL — no cifrado. Cualquiera puede decodificarlo y leer su contenido. La seguridad NO viene de que el payload sea secreto, sino de que el servidor rechaza cualquier token con firma inválida. El cliente puede leer el payload libremente para extraer exp u otros claims, pero nunca debe tomar decisiones de seguridad solo basándose en lo que lee — el servidor es el árbitro final.

🛟 Cápsula de rescate — decodificando Base64URL en la práctica

Si la IA te genera código de decodificación que no funciona en tu entorno, aquí está la solución mínima probada para cada plataforma. El error más común es que las LLMs sugieren APIs de Node.js (Buffer.from) que no existen en el browser, o ignoran las diferencias de caracteres entre Base64 estándar y Base64URL:

# ── Python — dos líneas ──────────────────────────────────────────────
import base64, json
def _decode_b64url(part: str) -> dict:
    # Base64URL → Base64 estándar: reemplazar - y _ , añadir padding
    padded = part.replace('-', '+').replace('_', '/') + '==' * 2
    return json.loads(base64.b64decode(padded))
# Nota: base64.b64decode ignora padding sobrante, así que '==' extra es seguro.

# ── JavaScript Browser — sin librerías externas ───────────────────────
function decodeB64URL(part) {
  // Paso 1: Base64URL → Base64 estándar
  const b64 = part.replace(/-/g, '+').replace(/_/g, '/');
  // Paso 2: Añadir padding hasta múltiplo de 4
  const padded = b64 + '='.repeat((4 - b64.length % 4) % 4);
  // Paso 3: Decodificar y parsear
  return JSON.parse(atob(padded));
  // ⚠️ atob() es browser-only. En Node.js usa Buffer.from(padded,'base64').toString()
}

🔒 Invariantes que no deben romperse — COMPRENDE

  • El payload del JWT no está cifrado — está codificado en Base64URL. Nunca almacenes datos sensibles (contraseñas, números de tarjeta) en el payload esperando que estén ocultos.
  • El cliente nunca verifica la firma del JWT. Si la IA te propone implementar HMAC o RSA en el cliente, está dando una solución incorrecta.
  • El campo exp usa Unix timestamp en segundos (no milisegundos). Un error de unidades hace que el cliente piense que el token expira en 1970 o en el año 2527.
  • Un JWT de 15 minutos no significa que el usuario deba re-autenticarse cada 15 minutos — el refresh_token (7 días) permite renovar el access_token sin credenciales. Son dos tokens con propósitos distintos.
  • Leer el claim role del JWT para mostrar/ocultar elementos de UI está bien. Usar ese claim para tomar decisiones de autorización en el cliente (ej: "el usuario es admin, así que proceso la solicitud") es incorrecto — el servidor es el árbitro de autorización.

💻 Lenguaje sugerido por ruta — ¿por qué importa la elección?

Ruta Python (terminal/servicio): requests.Session te da hooks (hooks['response']) para interceptar cada respuesta y disparar el refresh en 401. El manejo de concurrencia con asyncio.Lock hace visible el patrón singleton. Aprenderás cómo un objeto de sesión encapsula estado persistente entre peticiones.

Ruta Browser (JavaScript): Un wrapper sobre fetch() o un request interceptor demuestra el patrón Decorator — añades comportamiento (adjuntar token, manejar 401) sin modificar el código que llama a la API. El almacenamiento en memoria (variable de módulo) vs. sessionStorage tiene implicaciones reales de seguridad que analizarás en REFLEXIONA.

⚠️ Errores comunes esta semana

Error 1: Confundir `exp` en segundos con milisegundos. Date.now() en JavaScript devuelve milisegundos; time.time() en Python devuelve segundos (float). El claim exp del JWT siempre está en segundos Unix. Comparar Date.now() directamente con exp hace que el token parezca que expira en el año 2527 o que ya expiró en 1970.
Error 2: Múltiples refresh simultáneos — el "thundering herd" de tokens. Si 5 peticiones concurrentes reciben 401 al mismo tiempo, sin protección singleton dispararán 5 llamadas a /api/auth/refresh. Muchos servidores invalidan el refresh_token al primer uso (rotación de tokens) — las 4 peticiones restantes fallarán con 401, forzando logout. La solución es un flag o un Promise/Future compartido.
Error 3: Almacenar el refresh_token en localStorage sin considerar XSS. Un script malicioso inyectado en la página puede leer localStorage completo. El access_token en memoria (variable de módulo JS) limita la ventana de ataque a 15 minutos. El refresh_token es el token de mayor valor — su almacenamiento merece la mayor protección.
Error 4: No limpiar el timer de refresh proactivo al hacer logout. Si el usuario cierra sesión pero el setInterval/asyncio.create_task del refresh sigue corriendo, intentará hacer refresh con tokens ya eliminados — generando errores silenciosos o, peor, conservando el token activo en memoria después del logout.

Tu cliente calcula exp - time.time() para saber cuánto tiempo falta. Pero el reloj de tu máquina puede estar adelantado o atrasado respecto al servidor. Si tu cliente tiene el reloj 2 minutos adelantado, pensará que el token expiró 2 minutos antes de que el servidor lo invalide — causando refreshes innecesarios. Si está 2 minutos atrasado, intentará usar un token ya rechazado por el servidor. La solución práctica: usar un margen de expiración generoso (5 min) y siempre manejar el caso de 401 reactivo como la fuente de verdad definitiva.

OAuth 2.0 + PKCE — Cuando el cliente no maneja credenciales

En el flujo que implementaste esta semana, tu cliente recibe directamente las credenciales del usuario (email/password) y las envía al servidor de auth. Esto funciona cuando tu cliente es la aplicación que controlas. Pero, ¿qué pasa cuando EcoMarket quiere permitir que sus operadores se autentiquen con su cuenta de Google o Microsoft, sin que tu cliente vea su contraseña?

OAuth 2.0 Authorization Code con PKCE (Proof Key for Code Exchange) resuelve exactamente ese problema. El cliente nunca ve la contraseña del usuario — delega la autenticación al proveedor de identidad (Google, Microsoft, el propio servidor de auth de EcoMarket).

🔄 Token Rotation — la extensión natural del Reto 3

En la implementación de esta semana, el refresh_token puede usarse múltiples veces durante sus 7 días de vida. Algunos servidores implementan Token Rotation: cada vez que el cliente usa el refresh_token para obtener un nuevo access_token, el servidor devuelve también un nuevo refresh_token e invalida el anterior.

Ventaja: si alguien roba el refresh_token, el servidor lo detecta cuando el cliente legítimo intenta usarlo (el servidor verá dos intentos con el mismo token). Desventaja: si el cliente no guarda el nuevo refresh_token antes de que falle la conexión, el usuario queda sin poder renovar — debe re-autenticarse. Tu TokenManager necesitaría actualizar ambos tokens en el método refresh_access_token().

📖 Glosario — Semana 8

JWT (JSON Web Token)
Estándar (RFC 7519) para transmitir claims entre partes como un objeto JSON firmado digitalmente. Compuesto por header.payload.signature codificados en Base64URL.
Access Token
Token de corta duración (típicamente 15–60 min) que el cliente adjunta a cada petición autenticada. Su brevedad limita el daño si es interceptado.
Refresh Token
Token de larga duración (días o semanas) que el cliente usa exclusivamente para obtener un nuevo access_token sin re-autenticarse con credenciales.
Claim
Par clave-valor en el payload del JWT. Claims registrados por RFC 7519: sub, iss, aud, exp, iat, nbf, jti.
Base64URL
Variante de Base64 que usa - en lugar de +, _ en lugar de / y omite el padding =. Segura para URLs y headers HTTP.
Bearer Token
Esquema de autenticación HTTP donde el token se envía en el header Authorization: Bearer <token>. "Bearer" significa "quien porta este token tiene acceso".
Token Rotation
Estrategia donde cada uso del refresh_token genera un nuevo refresh_token e invalida el anterior, detectando uso doble del mismo token (posible robo).
XSS (Cross-Site Scripting)
Ataque donde un script malicioso se ejecuta en el contexto del navegador de la víctima. Puede leer localStorage y variables JS expuestas.
CSRF (Cross-Site Request Forgery)
Ataque donde una página maliciosa hace que el navegador de la víctima envíe una petición autenticada sin su conocimiento — aprovechando las cookies que el navegador adjunta automáticamente.
HttpOnly Cookie
Cookie con el flag HttpOnly: el navegador la envía automáticamente en peticiones HTTP pero el JavaScript de la página no puede leerla, eliminando el vector XSS.
PKCE (Proof Key for Code Exchange)
Extensión de OAuth 2.0 (RFC 7636) donde el cliente genera un code_verifier secreto y envía su hash (code_challenge) al servidor. Protege el authorization code contra interceptación.
Clock Skew
Diferencia de tiempo entre el reloj del cliente y el del servidor. Afecta el cálculo de expiración del token. Se mitiga con márgenes de refresh generosos y tratando 401 como fuente de verdad definitiva.
Refresh Singleton (AETL)
Patrón de implementación cliente donde solo una petición de refresh puede estar activa simultáneamente. Las demás peticiones concurrentes esperan el resultado de la primera en lugar de disparar peticiones adicionales.
Refresh Proactivo vs Reactivo (AETL)
Proactivo: el cliente anticipa la expiración leyendo el claim exp y renueva el token antes de que expire. Reactivo: el cliente espera el 401 del servidor para iniciar el refresh. El proactivo ofrece mejor experiencia de usuario.
Token Manager (AETL)
Componente cliente responsable del ciclo de vida completo de los tokens: almacenamiento, lectura de claims, refresh proactivo/reactivo, singleton de refresh y logout limpio. Abstracción que el interceptor HTTP usa sin conocer los detalles del JWT.
