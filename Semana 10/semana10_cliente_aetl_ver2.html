📋 Ver criterios de evaluación — Reto 1

A → ReceptorAlertas: parsea el stream SSE línea a línea y mantiene _last_event_id. Es el único componente que toca el protocolo SSE a nivel de bytes.

B → EventRouter: el método despachar() con diccionario de handlers es la firma del patrón Dispatcher. No conoce el protocolo de transporte.

C → TokenManager: decodificación Base64URL con padding restaurado — única responsabilidad del TM para extraer el payload del JWT.

D → CircuitBreaker: transición ABIERTO → SEMIABIERTO basada en tiempo, lanzando CircuitOpenError. Ningún otro componente tiene esta lógica de estados.

E → ClienteRobusto: orquesta sin duplicar — llama a is_expiring_soon() del TM, construye headers y pasa la lambda al CB. No reimplementa lógica de tokens ni de estados.

Error frecuente: asignar E al CircuitBreaker porque "ejecutar() está ahí". Pero el CB no construye headers ni llama a is_expiring_soon(). E orquesta; D decide.

# Síntomas arriba. Encuentra las 3 líneas problema.

import time, base64, json, logging
from enum import Enum

logger = logging.getLogger(__name__)

class EstadoCircuito(Enum):
    CERRADO = "CERRADO"; ABIERTO = "ABIERTO"; SEMIABIERTO = "SEMIABIERTO"

class CircuitBreaker:
    def __init__(self, umbral=5, timeout=60):
        self.estado = EstadoCircuito.CERRADO
        self._umbral = umbral; self._timeout = timeout
        self._fallos = 0;     self._tiempo_apertura = None

    def ejecutar(self, fn, token_manager=None):
        # ─────────────────────────────────────── BUG A (síntoma A)
        if token_manager:
            token = token_manager.get_access_token()
            pad = 4 - len(token.split('.')[1]) % 4
            payload = json.loads(
                base64.urlsafe_b64decode(token.split('.')[1] + '=' * pad)
            )
            if payload.get('rol') not in ('admin', 'supervisor'):
                raise PermissionError(f"Rol '{payload['rol']}' no autorizado")
        # ────────────────────────────────────────────────────────────
        if self.estado == EstadoCircuito.ABIERTO:
            if time.time() - self._tiempo_apertura >= self._timeout:
                self.estado = EstadoCircuito.SEMIABIERTO
            else:
                raise Exception("CircuitOpenError")
        try:
            resultado = fn()
            self._on_exito()
            return resultado
        except Exception as e:
            self._on_fallo(e); raise

    def _on_exito(self):
        # ─────────────────────────────────────── BUG C (síntoma C)
        self.estado = EstadoCircuito.CERRADO
        # FALTA: self._fallos = 0
        # ────────────────────────────────────────────────────────────

    def _on_fallo(self, error):
        if not self._es_fallo_servidor(error): return
        self._fallos += 1
        if self._fallos >= self._umbral:
            self.estado = EstadoCircuito.ABIERTO
            self._tiempo_apertura = time.time()

    def _es_fallo_servidor(self, error):
        msg = str(error).lower()
        return '5' in str(error) or 'timeout' in msg or 'connection' in msg

class TokenManager:
    def __init__(self): self._access_token = 'dummy.access.token'
    def get_access_token(self): return self._access_token
    def get_auth_header(self): return {'Authorization': f'Bearer {self._access_token}'}
    def is_expiring_soon(self): return False

class ClienteRobusto:
    def __init__(self, tm, cb): self._tm = tm; self._cb = cb

    async def get_inventario(self):
        headers = self._tm.get_auth_header()
        try:
            return await self._cb.ejecutar(
                lambda: self._mock_http_get('/api/inventario'),
                token_manager=self._tm     # ← relacionado con Bug A
            )
        except Exception as e:
            # ─────────────────────────────────── BUG B (síntoma B)
            logger.error(
                f"Error: {e}. Auth: {headers['Authorization'][:40]}..."
            )
            # ────────────────────────────────────────────────────────
            raise

    def _mock_http_get(self, path):
        raise ConnectionError("503 Service Unavailable")

📋 Ver criterios de evaluación — Reto 3

Bug A (SRP / INV-A1): CircuitBreaker.ejecutar() decodifica el JWT y verifica el rol. Violación de SRP: el breaker no debe conocer el contenido del token. Corrección: eliminar el bloque if token_manager: completo.

Bug B (Seguridad / INV-B2): headers['Authorization'][:40] imprime fragmentos del token en el log. Corrección: eliminar la interpolación del token — loguear solo el endpoint y el código de estado HTTP.

Bug C (INV-A3): _on_exito() cambia el estado a CERRADO pero no resetea self._fallos. Corrección: agregar self._fallos = 0 en _on_exito().

Puntuación: 7 pts por bug (diagnóstico + corrección + principio). Diagnóstico sin principio → 4 pts.

Árbitro de Reconexión SSE — ¿qué estrategia elige EcoMarket? 🚀 Bonus

+7 XP extra ⏱ fuera de taller

Cuando el Circuit Breaker está ABIERTO y la conexión SSE cae simultáneamente, hay 3 estrategias posibles. Evalúalas y argumenta cuál es correcta para EcoMarket en Black Friday.

Las 3 estrategias

A — Esperar al breaker: el ClienteSSEMultiplex no reconecta SSE hasta que el Circuit Breaker cierre (≥60 s).

B — SSE independiente: el ClienteSSEMultiplex reconecta inmediatamente — SSE es una conexión persistente distinta de las peticiones HTTP; el CB no la afecta.

C — Cache como fallback: mientras el breaker está abierto, el sistema sirve los últimos eventos SSE del caché local. No reconecta hasta que el breaker cierre.

📋 Ver análisis técnico — Reto 6

Estrategia B es técnicamente correcta cuando los servicios están separados: el CB (Semana 9) envuelve peticiones HTTP del ClienteRobusto, no la conexión SSE del ClienteSSEMultiplex. Son stacks distintos.

Trampa: si ambos servidores están en el mismo host, el CB abierto es evidencia de que el SSE también puede fallar — reconectar SSE contra un servidor caído genera tráfico innecesario. Un sistema robusto tendría un CB separado por protocolo.

📖 Glosario de Revisión — Semanas 6–9

Server-Sent Events (SSE)
Protocolo unidireccional (servidor→cliente) sobre HTTP. El cliente abre una conexión persistente; el servidor envía eventos en texto (data:, event:, id:). El cliente reconecta automáticamente con Last-Event-ID.
EventRouter
Componente del ClienteSSEMultiplex que despacha eventos SSE al handler registrado según el tipo. No conoce el protocolo de transporte ni la autenticación.
JWT — JSON Web Token
Token firmado: header.payload.signature en Base64URL. El cliente decodifica el payload sin verificar la firma. La verificación siempre ocurre en el servidor.
Base64URL
Variante de Base64 que reemplaza + por - y / por _, y omite el padding =. Los decodificadores estándar requieren restaurar el padding antes de procesar el payload JWT.
Circuit Breaker
Patrón del cliente que monitorea éxitos/fallos de llamadas remotas. Ante fallo sostenido, abre el circuito y rechaza peticiones inmediatamente para proteger al servidor durante su recuperación.
Estado Semiabierto (Half-Open)
Estado de transición: una sola petición de prueba pasa al servidor. Éxito → cierra (y resetea contador). Fallo → abre de nuevo con timeout completo reiniciado.
Fail Fast
Principio: fallar inmediatamente con un error claro es mejor que esperar el timeout completo. El CircuitBreaker abierto implementa fail fast devolviendo CircuitOpenError sin tocar la red.
Singleton de Refresh
Si múltiples peticiones concurrentes detectan el token expirado, solo una ejecuta el refresh. Las demás esperan al mismo resultado. Evita N peticiones redundantes al endpoint de autenticación.
INV-A / INV-B (AETL)
Familias de invariantes del curso. INV-A: del CircuitBreaker (no conoce JWT, una petición en SEMI, reset al cerrar, 401 no es fallo). INV-B: del TokenManager (no conoce el breaker, token fuera de logs, refresh singleton).
Deadlock Auth-Breaker (AETL)
Anti-patrón: el CB abierto bloquea las peticiones de refresh de token, dejando el cliente incapaz de autenticarse y de hacer peticiones simultáneamente. Se previene excluyendo el endpoint de auth del breaker.
ADR — Architecture Decision Record (AETL)
Documento breve que registra una decisión de diseño con su contexto, la decisión tomada y las consecuencias. Hace explícito lo que de otro modo quedaría implícito en el código.
Grand Deploy (AETL)
Denominación de la Semana 10 en la narrativa del curso: el momento en que todos los componentes se integran, auditan y certifican antes de "salir a producción" con el cliente EcoMarket.
